    # git 总结

    # 合并多次零散的提交

    前言: 有时候我们在开发一个功能的时候，可能会有多个提交记录，这个一来比较零散，同时当功能分支合并到发版分支的时候，可能会比较混乱，不利于日后的历史管理，我们可以采用git rebase -i来合并这些零散的提交

    git rebase -i [commitid](你要合并的提交记录的前一条commitid作为base)

# 会出现一个编辑器输入框,我们只要关注s即可，意思是将提交记录合并的上一个commit中

pick 32e3951 1:1
pick 803bfc5 1:2

# Rebase 1922107..803bfc5 onto 1922107 (2 commands)
#
# Commands:
# p, pick <commit> = use commit
# r, reword <commit> = use commit, but edit the commit message
# e, edit <commit> = use commit, but stop for amending
# s, squash <commit> = use commit, but meld into previous commit
# f, fixup <commit> = like "squash", but discard this commit's log message
# x, exec <command> = run command (the rest of the line) using shell
# d, drop <commit> = remove commit
# l, label <label> = label current HEAD with a name
# t, reset <label> = reset HEAD to a label
# m, merge [-C <commit> | -c <commit>] <label> [# <oneline>]

    我们将第二个提交的pick改成s即可

    pick 32e3951 1:1
s 803bfc5 1:2

    保存退出之后会弹出另外一个输入框，直接输出最终你需要的commit message就可以了

    # git worktree 使用

    前言： 在前端开发中，经常会遇到一些问题:

    1.我要切一个分支去更新 node_modules 的依赖，然后切回去的时候，node_modules 不做版本管理，但是还是需要原来的依赖,就又得重新 install 依赖，否则可能会依赖不一致导致程序报错,然后切回新分支又得重新 install，比较麻烦而已容易遗忘`

    2.又或者我在某个分支进行开发，临时要改别的分支的代码，用 git stash 还是比较麻烦，临时 commit 又没什么意义,而且容易出错

    针对上面的一些问题，最近无意间看到git worktree的用法，试了下能够很好的解决这个问题，特意总结一下:

    # git worktree

    默认情况下，git 仓库只有一个 worktree，就是 master 分支,可以通过git worktree list 来查看

    git worktree list
/Users/buzz/Code/demo/git/worktree  7473a35 [master]

    比如我现在要新做一个功能，升级 webpack2 到 webpack4,我可以如下操作:

    git worktree add feature-webpack4
## feature-webpack4 相当于一个path，默认就会在当前建立一个feature-webpack4的目录
## 同时多一个feature-webpack4分支

    之后我们就可以在 feature-webpack4 目录安装最新的依赖，这是一个独立的目录，我们直接可以 cd 进去安装最新的webpack4进行开发。

    cd feature-webpack4
yarn add webpack@4
git add .
git commit -m "update webpack4"

    如果想在原来的 webpack2 环境下开发，直接跳出 feature-webpack4 目录即可，又回到来了原来的 webpack2 的环境。 之后想要合并升级后的依赖，直接合并对应分支即可。

    cd ..
git merge feature-webpack4
yarn # 就会根据最新的依赖package.json来安装webpack4

    合并完成之后就可以通过git worktree remove feature-webpack4来清理 worktree,feature-webpack4分支可以手动清除。

    其他的一些指令可以通过 `git worktree --help`来获取更多的信息

    # Commit Message 格式

    对于update和fix bug等提交记录已经看的忍受不了，需要更加优雅,更加强制的提交方式。

    采用Commitizen和 cz-conventional-changelog 工具来约束和生成 Git Message

    # 安装

    npm i -g commitizen cz-conventional-changelog # 全局安装，有些需要sudo

    # 配置

    针对项目安装依赖 husky

    yarn add husky --dev

    配置 package.json

      "config": {
    "commitizen": {
      "path": "cz-conventional-changelog" # 配置预设的cz-conventional-changelog
    }
  },
  "husky": {
    "hooks": {
      "prepare-commit-msg": "exec < /dev/tty && git cz --hook" # 对于git commit 的钩子处理
    }
  }
